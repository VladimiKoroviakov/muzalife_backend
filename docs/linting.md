# Linting Guide — MuzaLife Backend

## Обраний лінтер

**ESLint v9+** (flat-config format)

### Чому ESLint?

ESLint є стандартом де-факто для статичного аналізу JavaScript та Node.js
проєктів. Основні причини вибору:

- **Широка екосистема** — тисячі готових правил і плагінів
- **Підтримка ES Modules** — проєкт використовує `"type": "module"` (ESM),
  ESLint повністю це підтримує
- **Конфігурація через JS** — `eslint.config.js` є звичайним модулем,
  що спрощує умовну логіку
- **Автоматичне виправлення** — більшість стилістичних помилок
  виправляються командою `--fix`
- **Інтеграція** — підтримується всіма популярними IDE, CI/CD системами
  та git-хуками

## Базові правила та їх пояснення

Конфігурація знаходиться у файлі `eslint.config.js`.

### Правила попередження помилок

| Правило | Рівень | Пояснення |
|---------|--------|-----------|
| `no-unused-vars` | error | Невикористані змінні засмічують код і є ознакою незавершеного рефакторингу |
| `no-undef` | error | Посилання на неоголошені змінні призводять до `ReferenceError` в runtime |
| `eqeqeq` | error | `==` виконує неявне приведення типів; `===` є надійнішим і прозорішим |
| `no-eval` | error | `eval()` є вектором атаки XSS та уповільнює виконання |
| `no-unreachable` | error | Недосяжний код вказує на логічну помилку |
| `no-constant-condition` | error | `if (true)` або `while (false)` — ознака залишкового дебагу |

### Правила якості коду

| Правило | Рівень | Пояснення |
|---------|--------|-----------|
| `no-var` | error | `var` має функціональну область видимості, що призводить до неочевидних багів; `let`/`const` безпечніші |
| `prefer-const` | error | Змінні, що не змінюються після ініціалізації, мають бути `const` |
| `prefer-template` | error | Шаблонні літерали читабельніші та безпечніші за конкатенацію рядків |
| `curly` | error | Обов'язкові фігурні дужки у всіх блоках запобігають помилкам при додаванні нових рядків |
| `object-shorthand` | error | `{ foo: foo }` → `{ foo }` — стислий синтаксис є стандартом ES6+ |

### Правила стилю

| Правило | Рівень | Пояснення |
|---------|--------|-----------|
| `no-console` | warning | `console.log` в production коді є поганою практикою; використовуйте логер |
| `semi` | error | Обов'язкові крапки з комою усувають неоднозначність ASI |
| `quotes` | error | Одиночні лапки — стандарт у Node.js-спільноті |
| `no-trailing-spaces` | error | Зайві пробіли в кінці рядків засмічують git diff |
| `eol-last` | error | Порожній рядок наприкінці файлу — POSIX-стандарт |
| `comma-dangle` | error | Trailing comma дозволяє чистіші git diff при додаванні елементів |

### Виключення для скриптів

Правило `no-console` вимкнено для файлів у директорії `scripts/`, оскільки
скрипти міграції бази даних та налаштування навмисно виводять інформацію
в консоль.

## Запуск лінтера

### Перевірка всього проєкту

```bash
npx eslint .
```

### Автоматичне виправлення

```bash
npx eslint . --fix
```

### Перевірка конкретного файлу або директорії

```bash
npx eslint controllers/authController.js
npx eslint routes/
```

### Виведення у форматі JSON

```bash
npx eslint . --format json -o lint-report.json
```

### Через npm-скрипти (після налаштування `package.json`)

```bash
npm run lint        # перевірка
npm run lint:fix    # виправлення
```

## Git Hooks

Pre-commit хуки запускають лінтер автоматично перед кожним комітом.
Детальніше — у розділі нижче.

### Налаштування через Husky + lint-staged

```bash
# Встановити Husky та lint-staged
npm install --save-dev husky lint-staged

# Ініціалізувати Husky
npx husky init
```

Додати до `package.json`:

```json
"lint-staged": {
  "*.js": ["eslint --fix", "git add"]
}
```

Вміст файлу `.husky/pre-commit`:

```sh
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"
npx lint-staged
```

Після коміту хук автоматично перевіряє і, де можливо, виправляє
лише ті файли, що входять до коміту.

## Інтеграція з процесом збірки

Лінтинг додано як окремий npm-скрипт та вбудовано в загальну команду
перевірки коду:

```json
"scripts": {
  "lint": "eslint .",
  "lint:fix": "eslint . --fix",
  "check": "npm run lint"
}
```

Команда `npm run check` є точкою входу для CI/CD-перевірок.
У GitHub Actions рекомендовано додати крок:

```yaml
- name: Lint
  run: npm run lint
```

## Статична типізація

Для Node.js/JavaScript рекомендується поступовий перехід на **TypeScript**.

### Кроки міграції

1. Встановити TypeScript:

   ```bash
   npm install --save-dev typescript @types/node
   ```

2. Ініціалізувати `tsconfig.json`:

   ```bash
   npx tsc --init
   ```

3. Додати `@typescript-eslint` плагін:

   ```bash
   npm install --save-dev @typescript-eslint/parser @typescript-eslint/eslint-plugin
   ```

4. Поступово перейменовувати файли з `.js` на `.ts` і виправляти типи.

Альтернатива без повної міграції — JSDoc-анотації з перевіркою через
`tsc --checkJs` у режимі `allowJs`.
